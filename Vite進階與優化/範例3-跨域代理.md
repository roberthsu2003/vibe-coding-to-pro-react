# 範例 3：跨域代理 (Development Proxy)

## 目標

解決前端開發階段呼叫後端 API 最常遇到的 `CORS policy` (跨來源資源共用) 錯誤。

---

## 步驟 1：了解跨域代理的原理

在開發階段，如果前端 (`localhost:5173`) 直接呼叫後端 (`localhost:3000`)，瀏覽器基於安全性會阻擋這個請求並拋出 CORS 錯誤。與其去改後端設定，我們可以利用 Vite 開發伺服器當作「代理人」：
1. 您向 Vite 發送 `/api/users` 請求。
2. Vite 偷偷將請求轉發給後端 `http://localhost:3000/api/users`。
3. 這樣對瀏覽器來說是同網域請求，完美避開 CORS 限制！

---

## 步驟 2：觀察 Vite Proxy 設定

開啟 `vite-advanced-demo/vite.config.ts`。

尋找 `server.proxy` 設定區塊：
```typescript
server: {
  proxy: {
    '/api': {
      target: 'http://localhost:3000', // 目標後端網址
      changeOrigin: true, // 必設：改變請求頭的 host，欺騙後端這是一個真請求
      rewrite: (path) => path.replace(/^\/api/, ''), // 將 /api 前綴移除 (視後端網址需求而定)
    },
  },
}
```
這個設定代表：當您在程式中發出 `/api/xxx` 的 API 請求時，Vite 會幫您轉交給 `http://localhost:3000/xxx` 處理。

---

## 步驟 3：在程式中使用代理路徑

開啟 `vite-advanced-demo/src/App.tsx` 觀察或嘗試在其他元件中的 API 呼叫。

原本會寫成：
```typescript
const response = await fetch('http://localhost:3000/api/data');
```
由於我們使用了代理，可以直接寫相對路徑：
```typescript
const response = await fetch('/api/data');
```

---

## 完成

透過 Proxy 設定，開發階段的 API 串接將充滿彈性且不受 CORS 阻擋。

> **💡 為何這招很重要？**
> 
> 這個技巧在串接 AI 應用時極為關鍵！為了保護 API Key 不外洩，我們通常會自己架設一個 **後端伺服器 (例如使用 Express 建立的 BFF 架構)**。屆時，我們就是利用這招 `Vite Proxy`，讓前端順利呼叫自己的 Express 後端，再由後端去安全地呼叫 Gemini API。詳細實作請見後續的 [BFF + Express API Proxy 架構](../BFF+Express_API_proxy/README.md) 教學。
